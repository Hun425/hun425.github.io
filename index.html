비욘드메디슨 성능 개선

배경

Clickless 저장소의 통합 테스트가 @SpringBootTest + Testcontainers + @DirtiesContext 조합으로 인해 컨텍스트 재기동이 반복 발생. 전체 테스트 실행에 429초 소요.

해결

병목이 테스트 로직이 아닌 컨텍스트 재기동 비용임을 baseline 측정으로 확인. 테스트 의미를 보존하면서 @DirtiesContext 대신 Redis cleanup + DB snapshot restore로 전환하여 컨텍스트 캐시 재사용을 극대화하는 방향을 선택.

결과

아이씨티웨이 성능 개선

배경

약 2,260만 건의 차량 등록 데이터 검색 시 PostgreSQL LIKE 쿼리로 인한 Full Table Scan 발생. 마지막 페이지 조회 시 응답 시간 최대 20초까지 소요.

해결

결과

Spring WebSocket을 활용한 실시간 퀴즈 배틀 기능 구현

팀 프로젝트 실시간 설계

배경

사용자 간 실시간 상호작용 학습 필요. 기존 HTTP 요청/응답 방식으로는 배틀 상태 동기화 불가. 폴링 방식은 트래픽 유발 및 실시간성 저하.

해결

폴링 방식의 불필요한 트래픽과 실시간성 한계를 해소하기 위해 양방향 통신이 가능한 WebSocket을 선택. STOMP 프로토콜로 구독/발행 패턴을 적용해 배틀방 단위 메시지 격리와 동시성 제어 구현.

결과

실시간 퀴즈 배틀 기능 구현 완료. 폴링 대비 서버/네트워크 부하 감소 및 실시간성 보장. WebSocket 기반 메시징 아키텍처 구축 경험 확보.

Saga 및 Outbox 패턴 기반의 분산 트랜잭션 구현

개인 프로젝트 아키텍처 설계

배경

여러 모듈에 걸친 비즈니스 로직에서 단일 ACID 트랜잭션으로 데이터 정합성 보장 불가. 처리 중 장애 발생 시 데이터 불일치 문제.

해결

모듈 간 강결합 없이 데이터 정합성을 보장하기 위해 2PC 대신 Saga(Orchestration)를 선택. 로컬 커밋과 이벤트 발행의 원자성 문제를 해결하기 위해 Outbox 패턴 + Debezium 조합으로 At-Least-Once 보장 설계.

결과

장애 전파 방지로 시스템 안정성·회복탄력성 향상. 모듈 간 결합도 감소, 유지보수성 및 확장성 확보.
