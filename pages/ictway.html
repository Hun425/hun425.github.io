기술적 의사결정

Kafka + Spark 분산 처리 아키텍처

배경: 전국 도로 구간별 차량 속도 데이터가 실시간으로 대량 유입되어, 단일 서버에서는 처리가 불가능한 상황이었습니다.

선택: Kafka로 데이터를 토픽별로 분리 수집하고, Spark의 병렬 처리 능력을 활용하여 구간별 집계를 수행했습니다.

결과: 수평 확장이 가능한 구조로, 데이터 증가에도 안정적인 처리량을 유지할 수 있게 되었습니다.

DB 비동기 인덱스 전략

배경: 대량 데이터 INSERT 시 인덱스 갱신으로 인한 병목이 발생하여, 저장 시간이 100분 이상 소요되었습니다.

선택: 데이터 적재 시 인덱스를 일시 제거하고, 적재 완료 후 비동기로 인덱스를 재생성하는 전략을 도입했습니다.

결과: 저장 성능 60% 최적화 (100분 → 40분)를 달성했습니다.

트러블 슈팅

Problem: 배치 처리 파이프라인 전체 소요 시간 10일

기존 시스템은 단일 프로세스로 순차 처리하여, 한 달치 데이터를 처리하는 데 10일이 소요되었습니다. 데이터 증가에 따라 처리 기한을 초과할 위험이 있었습니다.

Solution: Kafka 파티셔닝 + Spark 병렬 처리 + DB 비동기 인덱스 전략을 조합하여, 데이터 수집-가공-저장 전 구간을 최적화했습니다.

Result: 전체 배치 처리 시간 10일 → 4일로 약 60% 단축
